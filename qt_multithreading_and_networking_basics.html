C++ / Qt multithreading and networking basics

 Qt is very extensive and powerful framework based on C++ for wide range of applications, it's often recommended answer for questions like "What should I use to create GUI in C++?" or "Is there any good networking library for C++?". Important thing is, that with Qt we can create portable code and it has very good documentation and wide community always ready to help. Qt also offers development environment Qt Creator that is also great and recommended when working with Qt to get fully into this ecosystem.
I made the same decision when I have chosen to work with Qt for GUI design, so I decided to recreate WinHttp functionality with Qt Network library. My plan for this has been already hinted in WinHttp article to design application connecting with Binance cryptocurrency market and below representation is solid draft for fully functional multithread program with network solution.
 Because Qt is fully functional framework for wide range of purposes, its one of main classes is definitely QObject and it's certain that user finally bumps into it, as lots of classes inherit from it. With this subject is strictly connected macro Q_OBJECT, as per the documentation:
The Q_OBJECT macro must appear in the private section of a class definition that declares its own signals and slots or that uses other services provided by Qt's meta-object system.
Also QApplication class that we use in main.cpp is not an exception, its classes inheritance path goes like that:
QApplication -> QGuiApplication -> QCoreApplication -> QObject.
#include "binRaider.h" #include <QApplication> int main(int argc, char *argv[]) { QApplication a(argc, argv); binRaider w; w.show(); return a.exec(); }
We use QApplication class that provides all the functionality for QWidget-based applications but we just use it as the one to start the program with exec() function and before we use binRaider class which is the main one with show() function - this class inherits from QMainWindow in public mode and QMainWindow from QWidget - that's why we need to define QWidget pointer argument in constructor, but luckily Qt does it automatically for us, class destructor too. It's all visible in binRaider.h header below:

Also Ui namespace and Ui::binRaider *ui pointer are created for us when starting new project, it's very important because we can communicate with the GUI by using it and make all the changes or get GUI objects statuses and update variables. We have here not only basic initial variables, directives and declarations but I also declared my project stuff. We can see like mentioned before slots to which we get some data or just get them invoked (like because of the clicked button that sent the signal), as they're just void functions and we have signals we can send to GUI or to other slots inside the backend.
 The header file leads us to source, which is binRaider.cpp file. It expands all the content, including functions that manage multi-threaded tasks and use networking part, here is the short description of all of them:
- void binRaider::on_actionGetWallet_triggered() - it's the slot function invoked by the proper dropdown menu in GUI, it connects with Binance server through binCon::binConCl object, downloads required wallet data and updates proper part of the GUI by creating table with the data.
- void binRaider::on_pushButton_candlesPrint_clicked() - it's also invoked from GUI but by the button and connects to Binance to download klines for specified symbol, interval and their count to update the part of the GUI.
- void binRaider::on_pushButton_autobotStart_clicked() - this function is also invoked with proper button and starts a new thread. In function's scope we initialize mThread object dynamically with new QThread class object and also new worker as our binConThreadWorker class object that controls all the work in a new thread, these two things are essence of the Qt multithreading with the following QObject::connect function:
 - QObject::connect(mThread, SIGNAL(started()), worker, SLOT(process_autobotStart())) - we connect signal started() from the thread with our worker's slot process_autobotStart(). It's going to be sent after we invoke start() function on our mThread object.
 - QObject::connect(worker, SIGNAL(valueChanged(QString)), this, SLOT(onValueChanged(QString))),
   QObject::connect(worker, SIGNAL(valueSecChanged(QString)), this, SLOT(onValueSecChanged(QString))) - here we connect signals from our mThread with slots in the main class where we are (that's why we use this) that update GUI objects.
 - QObject::connect(worker, SIGNAL(finished()), mThread, SLOT(quit())),
   QObject::connect(worker, SIGNAL(finished()), worker, SLOT(deleteLater())),
   QObject::connect(mThread, SIGNAL(finished()), mThread, SLOT(deleteLater())) - we connect finished() signals to quit() and deleteLater() functions to safely finish our work with these objects and delete them when thread is finished. Finally then we can move our worker to the created mThread and start the new thread. It is worth to clarify that the whole program works in the main thread and there is event loop that constantly waits to receive and send signals but also do standard work we want like the one inside the functions. Saying it very simply it's kind of while infinite loop checking if any triggers have occured. When moving worker so object of our class to a new thread, we can run separate tasks there and we don't block the main thread. Additionally it's important to mention that GUI handling runs only in the main thread! User shouldn't try to access connection with GUI from separate thread.

This leads us to binConThreadWorker.h, I won't present the whole functionality in cpp file that can be arranged according to the developer's preferences. We have seen in binRaider.h the void onStartTrade(QString) function that receives signals from our new thread and we can see below corresponding signal void startTrade(QString). The purpose is to ask the main thread to start a next new thread that is going to do another task. The whole process path looks like this:
button in GUI is clicked -> (binRaider's class SLOT) void binRaider::on_pushButton_autobotStart_clicked(), when mThread is started, then started() SIGNAL is sent to binConThreadWorker -> (binConThreadWorker's class SLOT) void process_autobotStart(), after analyze SIGNAL void startTrade(QString) is sent to binRaider with symbol in QString variable -> (binRaider's class SLOT) void onStartTrade(QString) -> (binRaider's class SIGNAL) void on_process_startTrader1(QString) (in the first turn) is sent to binConThreadWorker's class SLOT -> void process_startTrader(QString).

We can notice important included classes with QNetwork prefix but also QUrl one and very meaningful QEventLoop.
Event loop is important design pattern in many programming languages that waits for and dispatches events or messages in a program, it works in the same thread but it's kind of next to it, it's not blocking the stack and it's especially noticable for long-term tasks, instead it queues the tasks and call them back to the stack when task is ready, for example asynchronous network communication. Same as in WinHttp, I use JSON for Modern C++. Also please remember to try use Qt classes like QString instead of std::string, otherwise use conversion functions like QString::fromStdString(std::string).  In the internet we can find a few HTTP request patterns in Qt, some of them are considered bad practice, however I think for simple projects and for specific purposes they're enough. Qt generally bases on signals and slots pattern and this should be the correct way to design it, to separate algorithm to more individual steps. Let's take a look at json getCandles(std::string coin, std::string interval, std::string limit) function and first example with QEventLoop:


json binConCl::getCandles(std::string coin, std::string interval, std::string limit) { QObject::connect(&mgr, SIGNAL(finished(QNetworkReply*)), &eventLoop, SLOT(quit())); std::string endpoint_candles = "/api/v3/klines"; std::string url = burl + endpoint_candles + "?symbol=" + coin + "USDT&interval="+interval+"&limit="+limit; req.setUrl(QUrl(QString::fromStdString(url))); reply = mgr.get(req); eventLoop.exec(); buffer = reply->readAll(); delete reply; std::string buffer_s = buffer.toStdString(); try { j = json::parse(buffer_s); } catch (const json::parse_error& e) { QMessageBox msgBox; msgBox.setText("Message: " + QString::fromStdString(e.what()) + ".\nException id: " + QString::number(e.id) + ".\nByte position of error: " + QString::number(e.byte) + ".\nPossibly no data has been downloaded and buffer is empty."); msgBox.setWindowTitle("Internal binCon Error!"); msgBox.exec(); } return j; }
The most important part is when we send the request reply = mgr.get(req), because we start then asynchronous operation. If eventLoop.exec() would be omitted, then we wouldn't have guarantee that buffer = reply->readAll() will be completed, as the reply can not come yet! With eventLoop.exec() we block the stack and kind of give the priority to execute all event loop tasks and then continue with rest on the stack, when reply is finished and QEventLoop gets the information to quit, thanks to QObject::connect(&mgr, SIGNAL(finished(QNetworkReply*)), &eventLoop, SLOT(quit())).
 The second option that does the stuff is QApplication::processEvents(), however as mentioned in the documentation:
Use of this function is discouraged. Instead, prefer to move long operations out of the GUI thread into an auxiliary one and to completely avoid nested event loop processing. If event processing is really necessary, consider using QEventLoop instead.
We need to #include <QApplication> also to make it works:


json binConCl::getCandles(std::string coin, std::string interval, std::string limit) { std::string endpoint_candles = "/api/v3/klines"; std::string url = burl + endpoint_candles + "?symbol=" + coin + "USDT&interval="+interval+"&limit="+limit; req.setUrl(QUrl(QString::fromStdString(url))); reply = mgr.get(req); while (!reply->isFinished()){ QApplication::processEvents(); } buffer = reply->readAll(); delete reply; std::string buffer_s = buffer.toStdString(); try { j = json::parse(buffer_s); } catch (const json::parse_error& e) { QMessageBox msgBox; msgBox.setText("Message: " + QString::fromStdString(e.what()) + ".\nException id: " + QString::number(e.id) + ".\nByte position of error: " + QString::number(e.byte) + ".\nPossibly no data has been downloaded and buffer is empty."); msgBox.setWindowTitle("Internal binCon Error!"); msgBox.exec(); } return j; }
We do not use any signals/slots in this example, just process the event loop taska until reply is finished. By the way it comes to my mind wrong code, that is not going to work, but I met a few times questions about it and I concidered to use it myself, it looks exactly the same like above, just without QApplication::processEvents() inside the while loop. In such case we run the empty loop and expect it will finish after it gets reply and allow to read all data. It won't happen as it will completely block the program and won't allow to process event loop and get the response back to the stack.
 The last pattern I want to mention is the most correct one that fits best Qt philosphy if I may say so:


void binConCl::replyFinished(QNetworkReply *reply) { buffer = reply->readAll(); delete reply; std::string buffer_s = buffer.toStdString(); try { j = json::parse(buffer_s); } catch (const json::parse_error& e) { QMessageBox msgBox; msgBox.setText("Message: " + QString::fromStdString(e.what()) + ".\nException id: " + QString::number(e.id) + ".\nByte position of error: " + QString::number(e.byte) + ".\nPossibly no data has been downloaded and buffer is empty."); msgBox.setWindowTitle("Internal binCon Error!"); msgBox.exec(); } emit getCadlesReadySignal(j); } void binConCl::getCandles(std::string coin, std::string interval, std::string limit) { QObject::connect(&mgr, SIGNAL(finished(QNetworkReply*)), this, SLOT(replyFinished(QNetworkReply*))); std::string endpoint_candles = "/api/v3/klines"; std::string url = burl + endpoint_candles + "?symbol=" + coin + "USDT&interval="+interval+"&limit="+limit; req.setUrl(QUrl(QString::fromStdString(url))); reply = mgr.get(req); }
First we connect signal from reply to slot in our class, that's why we pass this replyFinished(QNetworkReply*). I had some issue at first when testing this solution, because QObject::connect() as syntax suggest- it connects QObjects. I didn't have any need to make my class binConCl inherits from QObject, but in such case it's necessary (and remember about Q_OBJECT macro!):
namespace binCon { class binConCl: public QObject { Q_OBJECT //(...)
All the job for getCandles function is done when we ask for the response and void binConCl::replyFinished(QNetworkReply *reply) will handle the reading. Finally, we can emit the signal exactly like we did earlier with built-in signals in Qt classes, the same what applies to GUI for example. Of course we have to declare proper QObject connection so emit signal goes to proper slot and declare it in binCon.h. In the end I would like to advise to always write code in separate .cpp and .h files to avoid problems like I did bump into when not doing it, Qt program just couldn't compile the program, complained that is not possible to generate moc files etc., after the change everything "just worked" ;).
